The problem

Health software needs governance that is architectural, not decorative.

Health applications frequently need to coordinate identity, consent, professional habilitation, clinical purpose, operational data, drafts, finality, and audit. Treating those duties as scattered application logic creates brittle boundaries and unclear authority.

HealthOS approaches the problem as infrastructure: applications are governed consumers, while law-like controls remain in Core and mediated runtime surfaces instead of being reimplemented inside every Stage.

The solution thesis

Separate application ambition from juridical authority.

HealthOS defines a platform hierarchy where Core provides constitutional law, runtimes execute mediated work, Boundary exposes app-safe surfaces, and Stages consume those surfaces without becoming law engines.

Tier 1

Mestral Core

Consent, habilitation, storage law, provenance, gate, finality, and audit as platform-level rules.

Tier 2

GOS & Runtimes

Session Runtime, AACI, MSR, Async, User-Agent, Service Runtime, and governed provider seams.

Tier 3

Custom Boundary

Facades, envelopes, safe references, mediated state, degraded state, and command/result contracts.

Tier 4

Stages

Scribe, Veridia, and CloudClinic as governed application consumers, not constitutional authorities.

FAQ

Clear answers for external stakeholders.

What is HealthOS?

HealthOS is a scaffold/foundation-stage juridical execution environment for health applications. Its thesis is that governed health software should run under shared platform law instead of duplicating legal and operational controls inside each app.

Who is it for?

The repository positions HealthOS for governed health applications and their operators: professional-facing, patient-facing, service-operations, runtime, provider, and construction-system surfaces.

What is available today?

The repository includes Core contracts, runtimes, Boundary contracts, Stage scaffolds, a CLI first-slice path, schemas, TypeScript construction tooling, SQL shape, docs, tests, and validation scripts.

Is it production-ready?

No. The repository explicitly describes the project as scaffold/foundation-stage work. It is not a production system, not a complete EHR, and not a real regulatory/provider integration.

How can investors or partners engage?

Start with the repository and architecture docs, then request a technical walkthrough focused on the platform thesis, executable first slice, contract model, Stage boundaries, and open gaps.

Next step

Evaluate the foundation. Challenge the architecture. Inspect the claims.

HealthOS is presented here as an ambitious but honest foundation: strong enough to evaluate, early enough to shape, and explicit about what is not yet proven.
