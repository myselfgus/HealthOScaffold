# AI coding behavior contract

## Before coding

An AI must read, in order:
1. `README.md`
2. `docs/execution/README.md`
3. `docs/execution/00-master-plan.md`
4. `docs/execution/01-agent-operating-protocol.md`
5. current relevant `todo/` file
6. relevant architecture file(s)

## Prohibited behaviors

- coding directly into app layer to compensate for undefined core law
- adding remote/public access by convenience
- bypassing gate mechanics
- storing raw direct identifiers in casual operational payloads
- choosing provider-specific hacks in shared contracts
- skipping status updates after work completion

## Required behaviors

- define or refine contracts before implementation when ambiguity exists
- document any ontology-affecting decision
- keep human/computational plane separation intact
- preserve service isolation and user/service role distinctions
- keep AACI bounded to automation, drafting, retrieval, structuring, and proposal

## Completion discipline

After a coding task, the AI must:
- update status tracking
- update the corresponding todo file
- state what became unblocked
- state what remains risky or unresolved

## Review questions an AI must ask itself

- did I accidentally move law into UI?
- did I accidentally let AACI act as clinician?
- did I accidentally erase the distinction between identifiers and operational content?
- did I accidentally bypass consent/habilitation/provenance?
- did I accidentally make future mesh expansion harder by hardcoding local assumptions into ontology?
