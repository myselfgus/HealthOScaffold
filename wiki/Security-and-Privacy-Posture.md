# Security and Privacy Posture

> HealthOS is designed around governed health operations. Security and privacy are not side features; they are execution constraints.

---

## Current status

This page describes the HealthOS scaffold/foundation posture. It is not a production security certification and must not be read as proof of production readiness.

HealthOS currently emphasizes:

- consent-governed processing;
- habilitation-aware professional action;
- lawful context requirements;
- draft/final separation;
- mandatory gate before regulatory effectuation;
- provenance and audit semantics;
- fail-closed behavior;
- explicit maturity and production-gap language.

Production security still requires hardened deployment, access control, key management, incident response, monitoring, backup/restore automation, retention workflows, and operational review.

---

## Security model principles

| Principle | Meaning |
| :--- | :--- |
| Fail closed | Missing authority, consent, lawful context, or capability should deny or degrade, not silently proceed. |
| Least authority | Applications consume mediated surfaces; they do not own constitutional law. |
| Clinical finality gate | AI and automation may draft; human gate controls finalization and effectuation. |
| Provenance by design | Clinical artifacts should carry lineage, actor, source, and transformation context. |
| Auditability | Governed actions should be reviewable after execution. |
| Data separation | Directly identifying data and operational content should remain strongly separated and governed. |
| Honest degradation | Unavailable providers, models, retrieval, or capabilities should be visible to users and callers. |

---

## Privacy posture

HealthOS is not zero-knowledge against its own Core. The Core must see enough operational context to apply law, enforce consent, mediate runtimes, and produce governed behavior.

The privacy objective is not to make governance blind. The objective is to ensure that identifying data, operational content, reidentification, and clinical effects are separated, minimized, governed, audited, and purpose-bound.

---

## Consent and finality

Consent is not only metadata. It is an execution precondition.

Finality is not a UI button. It is a governed state transition.

| Concept | Required posture |
| :--- | :--- |
| Consent | Must include clinical purpose/finality and govern processing. |
| Habilitation | Must be validated before professional clinical action. |
| Draft | May be generated by runtime or AI under governance. |
| Gate | Required before regulatory effectuation. |
| Final artifact | Must have gate resolution and provenance. |
| Rejected artifact | Must remain withheld and auditable. |

---

## AI and provider safety

AI capabilities must remain subordinate to Core Law and Boundary mediation.

Required behaviors:

- model/provider capability routing;
- fail-closed remote policy where applicable;
- visible degraded/unavailable state;
- provenance for generated artifacts;
- human review before final clinical effectuation;
- no hidden escalation from draft to final;
- no provider integration that bypasses ModelGovernance.

---

## Stage isolation

Stage applications must consume HealthOS through Boundary surfaces. They should not import Tier 1 or Tier 2 modules directly.

This protects:

- legal consistency;
- app isolation;
- reviewability;
- runtime substitution;
- Custom-defined capability/prohibition enforcement;
- patient and professional sovereignty.

---

## Production gaps to close

Before production claims, HealthOS needs hardened answers for:

| Area | Needed work |
| :--- | :--- |
| Identity and access | Multi-actor workflow, RBAC/ABAC details, access reviews, session policy. |
| Key management | Production key custody, rotation, recovery, hardware-backed policy. |
| Storage | Distributed storage hardening, backup/restore, retention, export, deletion workflows. |
| Network/fabric | Sovereign mesh implementation, ACL tooling, monitoring, operator runbooks. |
| Observability | Security telemetry, audit review workflows, alerting, incident evidence capture. |
| Incident response | Runbooks, escalation, containment, recovery, postmortem procedure. |
| Provider integrations | Real provider onboarding without weakening fail-closed and governance requirements. |
| Regulatory interoperability | Real endpoint/signature/effectuation integrations with role approval workflows. |
| CI/CD | Build/test/docs/security gates on PR and release. |

## Recommended GitHub files

The repository should include or maintain:

- `SECURITY.md` for vulnerability reporting and security scope;
- `.github/PULL_REQUEST_TEMPLATE.md` with governance checklist;
- `.github/ISSUE_TEMPLATE/security_concern.md` for non-sensitive security posture concerns;
- `.github/ISSUE_TEMPLATE/compliance_gap.md` for governance and regulatory gaps;
- `docs/security/` or canonical security docs under `HealthOS/Shared/docs`.

Sensitive vulnerabilities should not be filed as public issues.

---

## Operational rule

> A HealthOS feature is not secure because it is hidden. It is secure when authority, purpose, finality, provenance, denial, and audit are explicit and enforceable.
